## Lesson N: eval before deploy, finally

At once, I tried Gemini 2.5 Flash Lite, instead of Gemini 2.5 Flash, for event extraction. The results were noticeably worse — hallucinated lineups, wrong dates, events invented from non-event posts. Since I didn't have an eval set at all until recently, I just did a yolo run, and had to fix the result with regrets. Then there was Gemini 3.0 Flash and 3.1 Flash Lite Preview, and this time I still wanted to try, but with more rigorous eval. It's obvious I need it anyway soon and will need to move on from the 2.5 Flash model. 

I ended up building eval GUIs — one for venue profiles, one for event extraction. I spent quite long optimizing the UX and hotkeys for these, to make the annotation process smoother. Using the UI, I manually annotated 100+ samples (Instagram posts, profiles) to evaluate the models. 

A big bonus of this process was that it forced me to have a well-defined expectation: because I didn't want to waste my time, I had to generate good "ground truth" that would work for both precision and recall.

Now that I have a full suite, I still think it was correct to not have an eval at day 1. Eval requires a well-defined expectation, which requires a clear product spec, which you can only mature through actually building and using the product. Also, how can you tell if your product is good enough to justify the eval effort?


## Now I use Gemini (API) and Qwen (local)

The pipeline now uses two different models with different roles:

**Gemini** (API) handles event extraction from Instagram posts — it needs to understand images, captions, and sometimes overlay text, all at once, precisely. Qwen 3.5 9B is not good enough for this, it turned out. So I still contribute to Google's revenue about.. under $100/month.

**Qwen 3.5 (9B)** (local, via Ollama) handles other smaller tasks. For example, musician classification — is this Instagram account actually a Korean jazz musician? This is a text-plus-image scoring task that doesn't need Gemini's ceiling, and I'm using my now 4-year old Macbook Pro (M1) happily for this. Another task is merging event information from multiple posts of the same event. After clarifying the problem into a smaller, comparison of the same field (e.g. "date", "time", "description"), Qwen handles this pretty well. I believe only by Qwen 3.5 this becomes easily solvable by a small, local LLM. The tasks that we call "simple" -- wasn't simple at all until very recently, really.

## Musician DB (WIP): the data shape

The musician pipeline was the biggest new build this cycle. The schema looks like this:

```
musicians
  ig_handle, locality, genre
  name_ko, name_en
  instruments, subgenre
  qwen_score            ← 1-10 from local Qwen classifier
  is_musician, is_korean, is_jazz   ← from manual annotation (golden set)

musician_venue_connections
  musician <> venue
  event_appearances     ← from events DB
  caption_mentions      ← from musician's own IG posts
  evidence              ← 'bidirectional' | 'venue_only' | 'musician_only'

musician_musician_connections
  musician_a <> musician_b
  co_appearance_count
```

The strongest evidence for a musician-venue connection is "bidirectional": the venue's event post mentions the musician, and the musician's own posts mention the venue. 44 of 369 connections currently have that level of confidence.

Current state: 102 musicians, 369 musician-venue connections, 137 musician-musician connections. The classifier promotion threshold is Qwen score ≥ 7, set empirically. The golden set eval will validate it.

## Giving the LLM a constitution

Everyone knows about `.claude` files, `CLAUDE.md`, memory systems. The practical reason is obvious: LLMs forget everything between sessions, so you write down what they need to know to resume.

I'd like to see it in a more fundamental way. Don't forget, no matter how surprising LLMs are, they're causal language models. They take `x`, some texts inputs, and produce `y`, the next tokens, their answers, or tool calling, thinking tokens, whichever you see. In setting up `.claude` files, what you're doing is constructing the input `x` for the LLM so that the output `y` is what you actually want. The problem is that sometimes, most of what shapes a good output is *unsaid* — things so obvious to you that you wouldn't think to write them down. "Don't silently swallow errors." "Precision over recall." "We'd rather show 80 musicians we trust than 200 we don't." No one writes these things down in a normal coding session. And yet they're exactly what you want an LLM collaborator to embody.

So that's what `PRINCIPLES.md` and the `org/` docs are in this repo. Not really documentation, but a constitution - mission, values, engineering principles, product principles, department-level docs — engineering, marketing — that describe personality and constraints.

In real world, for humans, these things are what define a company's culture. A mission statement, a set of values, a personality — all of that exists so that every decision-maker (human or otherwise) is working from the same set of unstated priors. When it's just you and an LLM, you're still a company, and these abstract concepts should set the right assumptions.
